# 并发编程

## Goroutine

Goroutine 是 Go 语言中轻量级的线程，由 Go 运行时管理。创建 Goroutine 非常简单，只需在函数调用前加上`go`关键字。

### 基本使用

```go
package main

import (
    "fmt"
    "time"
)

func sayHello(name string) {
    for i := 0; i < 3; i++ {
        fmt.Printf("Hello %s - %d\n", name, i)
        time.Sleep(100 * time.Millisecond)
    }
}

func main() {
    // 同步调用
    fmt.Println("=== 同步调用 ===")
    sayHello("同步")

    // 异步调用（Goroutine）
    fmt.Println("\n=== 异步调用 ===")
    go sayHello("Goroutine1")
    go sayHello("Goroutine2")

    // 等待Goroutine执行完成
    time.Sleep(1 * time.Second)
    fmt.Println("主程序结束")
}

/* 运行结果：
=== 同步调用 ===
Hello 同步 - 0
Hello 同步 - 1
Hello 同步 - 2

=== 异步调用 ===
Hello Goroutine1 - 0
Hello Goroutine2 - 0
Hello Goroutine1 - 1
Hello Goroutine2 - 1
Hello Goroutine1 - 2
Hello Goroutine2 - 2
主程序结束

【输出结果详解】
同步 vs 异步的区别：

同步调用：
- sayHello("同步") 直接调用函数，必须等待函数完全执行完才继续
- 按顺序打印 0 → 1 → 2，总耗时约 300ms (3次 × 100ms)

异步调用（Goroutine）：
- go sayHello() 创建新的goroutine并立即返回，不等待执行完成
- Goroutine1 和 Goroutine2 并发执行，几乎同时开始
- 两个goroutine交替打印（0-0, 1-1, 2-2），因为它们同时运行
- 总耗时约 300ms（并发执行，不是 600ms）

关键知识点：
- 不加 go：顺序执行，阻塞当前流程
- 加 go：并发执行，立即返回继续执行下一条语句
- time.Sleep(1 * time.Second)：等待goroutine执行完成，否则主程序会立即退出
- Goroutine非常轻量，可以轻松创建成千上万个
*/
```

### Goroutine 的特点

```go
package main

import (
    "fmt"
    "runtime"
    "sync"
    "time"
)

func main() {
    // 查看当前goroutine数量
    fmt.Printf("初始Goroutine数量: %d\n", runtime.NumGoroutine())

    // 创建多个Goroutine
    for i := 0; i < 5; i++ {
        go func(id int) {
            fmt.Printf("Goroutine %d 开始执行\n", id)
            time.Sleep(2 * time.Second)
            fmt.Printf("Goroutine %d 执行结束\n", id)
        }(i)
    }

    fmt.Printf("创建Goroutine后数量: %d\n", runtime.NumGoroutine())

    // 等待所有Goroutine执行完成
    time.Sleep(3 * time.Second)
    fmt.Printf("等待后Goroutine数量: %d\n", runtime.NumGoroutine())
}

/* 运行结果：
初始Goroutine数量: 1
Goroutine 0 开始执行
Goroutine 1 开始执行
Goroutine 2 开始执行
Goroutine 3 开始执行
Goroutine 4 开始执行
创建Goroutine后数量: 6
Goroutine 0 执行结束
Goroutine 1 执行结束
Goroutine 2 执行结束
Goroutine 3 执行结束
Goroutine 4 执行结束
等待后Goroutine数量: 1

【输出结果详解】
Goroutine数量的变化分析：

初始状态：
- 初始Goroutine数量: 1（只有main函数的goroutine）

创建5个Goroutine后：
- 创建Goroutine后数量: 6（1个主goroutine + 5个新创建的goroutine）
- 5个goroutine几乎同时打印"开始执行"
- 它们都进入2秒睡眠状态

等待3秒后：
- 所有goroutine的2秒睡眠已结束，打印"执行结束"
- 等待后Goroutine数量: 1（5个工作goroutine已结束，只剩主goroutine）

关键知识点：
- runtime.NumGoroutine() 可以查看当前存活的goroutine数量
- go func(id int) {...}(i) 是匿名函数的立即调用，(i)将参数传递给函数
- 必须传递参数(i)，否则所有goroutine会共享同一个i变量（闭包陷阱）
- Goroutine非常轻量，每个只占用约2KB栈空间（线程通常需要1-2MB）
- 主goroutine退出时，所有其他goroutine会被强制终止
*/
```

## Channel

Channel 是 Goroutine 之间通信的管道，可以安全地在多个 Goroutine 之间传递数据。

### 基本 Channel 操作

```go
package main

import (
    "fmt"
    "time"
)

func main() {
    // 创建无缓冲channel
    ch := make(chan string)

    // 发送数据到channel
    go func() {
        time.Sleep(1 * time.Second)
        ch <- "Hello from Goroutine"
    }()

    // 从channel接收数据
    msg := <-ch
    fmt.Println("接收到:", msg)

    // 创建带缓冲的channel
    bufferedCh := make(chan int, 3)

    // 向缓冲channel发送数据
    bufferedCh <- 1
    bufferedCh <- 2
    bufferedCh <- 3
    // bufferedCh <- 4 // 这里会阻塞，因为缓冲区已满

    // 从缓冲channel接收数据
    fmt.Println("从缓冲channel接收:", <-bufferedCh)
    fmt.Println("从缓冲channel接收:", <-bufferedCh)
    fmt.Println("从缓冲channel接收:", <-bufferedCh)
}

/* 运行结果：
接收到: Hello from Goroutine
从缓冲channel接收: 1
从缓冲channel接收: 2
从缓冲channel接收: 3

【输出结果详解】
无缓冲 vs 带缓冲 Channel：

无缓冲Channel（ch := make(chan string)）：
- ch <- "Hello" 发送操作会阻塞，直到有接收者
- msg := <-ch 接收操作会阻塞，直到有发送者
- 必须有配对的发送和接收操作，否则会死锁
- 主goroutine在 msg := <-ch 处阻塞1秒，等待数据

带缓冲Channel（bufferedCh := make(chan int, 3)）：
- 缓冲区大小为3，可以存储3个元素
- 发送操作：缓冲区未满时不阻塞，满了才阻塞
- 接收操作：缓冲区有数据时不阻塞，空了才阻塞
- 连续发送3个值(1,2,3)不会阻塞
- bufferedCh <- 4 会阻塞（如果取消注释），因为缓冲区已满

执行流程：
1. 创建无缓冲channel ch
2. 启动goroutine，1秒后向ch发送数据
3. 主goroutine在 <-ch 阻塞等待，收到数据后打印
4. 创建容量为3的缓冲channel
5. 连续发送3个值，不阻塞（缓冲区够用）
6. 依次接收3个值并打印

关键知识点：
- 无缓冲channel实现同步通信，强制goroutine配对
- 带缓冲channel实现异步通信，解耦发送和接收
- channel是goroutine之间通信的首选方式（不要通过共享内存来通信）
*/
```

### Channel 的方向

```go
package main

import "fmt"

// 只发送channel
func sender(ch chan<- string) {
    ch <- "Hello"
    ch <- "World"
    close(ch) // 关闭channel
}

// 只接收channel
func receiver(ch <-chan string) {
    for msg := range ch { // 使用range接收直到channel关闭
        fmt.Println("接收到:", msg)
    }
}

func main() {
    ch := make(chan string)

    go sender(ch)
    receiver(ch)
}

/* 运行结果：
接收到: Hello
接收到: World

【输出结果详解】
Channel方向的作用：

只发送Channel（chan<- string）：
- sender函数只能向channel发送数据，不能接收
- 编译时类型检查，防止误用
- ch <- "Hello" 合法，<-ch 会编译错误

只接收Channel（<-chan string）：
- receiver函数只能从channel接收数据，不能发送
- for msg := range ch 循环接收，直到channel关闭
- <-ch 合法，ch <- "xx" 会编译错误

执行流程：
1. main创建双向channel
2. 启动sender goroutine，发送"Hello"和"World"，然后关闭channel
3. receiver在主goroutine中执行，循环接收直到channel关闭
4. close(ch)后，range循环自动退出

关键知识点：
- 双向channel可以隐式转换为单向channel，反之不行
- chan<- 表示只写，<-chan 表示只读
- 关闭channel后，接收操作会立即返回零值，第二个返回值为false
- range会自动检测channel关闭，优雅退出循环
- 只有发送者应该关闭channel，接收者不应该关闭
- 向已关闭的channel发送数据会panic
*/
```

### Select 语句

Select 用于在多个 channel 操作中进行选择，类似于 switch 语句。

```go
package main

import (
    "fmt"
    "time"
)

func main() {
    ch1 := make(chan string)
    ch2 := make(chan string)

    go func() {
        time.Sleep(1 * time.Second)
        ch1 <- "来自channel1的消息"
    }()

    go func() {
        time.Sleep(2 * time.Second)
        ch2 <- "来自channel2的消息"
    }()

    // 使用select等待多个channel
    for i := 0; i < 2; i++ {
        select {
        case msg1 := <-ch1:
            fmt.Println("接收到:", msg1)
        case msg2 := <-ch2:
            fmt.Println("接收到:", msg2)
        case <-time.After(3 * time.Second):
            fmt.Println("超时!")
            return
        }
    }
}

/* 运行结果：
接收到: 来自channel1的消息
接收到: 来自channel2的消息

【输出结果详解】
Select语句的工作原理：

执行时间线：
- 0秒：启动两个goroutine，分别在1秒和2秒后发送消息
- 0秒：进入第一次select，所有case都阻塞，等待...
- 1秒：ch1有数据到达，case msg1 := <-ch1匹配，打印消息1
- 1秒：进入第二次select，继续等待...
- 2秒：ch2有数据到达，case msg2 := <-ch2匹配，打印消息2
- 2秒：循环2次结束，程序退出

Select特性：
- 类似switch，但专门用于channel操作
- 同时等待多个channel操作
- 哪个case先准备好，就执行哪个
- 如果多个case同时准备好，随机选择一个执行
- case <-time.After(3秒) 是超时机制，3秒内没收到消息就退出
- 如果没有default且所有case都阻塞，select会等待

关键知识点：
- select会阻塞，直到某个case可以执行
- time.After() 返回一个channel，指定时间后发送当前时间
- 超时机制防止永久阻塞
- 可以添加default分支实现非阻塞select
- select是处理多路复用的标准方式

常见应用场景：
- 等待多个goroutine的结果
- 实现超时控制
- 非阻塞的channel操作
- 优雅退出机制
*/
```

## 同步机制

### WaitGroup

WaitGroup 用于等待一组 Goroutine 执行完成。

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

func worker(id int, wg *sync.WaitGroup) {
    defer wg.Done() // Goroutine结束时调用Done

    fmt.Printf("Worker %d 开始工作\n", id)
    time.Sleep(time.Duration(id) * time.Second)
    fmt.Printf("Worker %d 工作完成\n", id)
}

func main() {
    var wg sync.WaitGroup
    // 启动多个worker
    for i := 1; i <= 3; i++ {
        wg.Add(1) // 增加等待计数
        go worker(i, &wg)
    }

    wg.Wait() // 等待所有worker完成
    fmt.Println("所有worker已完成")
}

/* 运行结果：
Worker 1 开始工作
Worker 2 开始工作
Worker 3 开始工作
Worker 1 工作完成
Worker 2 工作完成
Worker 3 工作完成
所有worker已完成

【输出结果详解】
为什么是这样的执行顺序？

执行时间线：
- 0秒：3个goroutine几乎同时启动，立即打印"开始工作"
- 1秒：Worker 1睡眠1秒后完成（time.Sleep(1 * time.Second)）
- 2秒：Worker 2睡眠2秒后完成（time.Sleep(2 * time.Second)）
- 3秒：Worker 3睡眠3秒后完成（time.Sleep(3 * time.Second)）
- 3秒：wg.Wait()解除阻塞，主程序继续执行，打印"所有worker已完成"

关键知识点：
- go关键字创建并发执行的goroutine，它们几乎同时启动
- time.Sleep(time.Duration(id) * time.Second)：每个worker睡眠不同时长
- defer wg.Done()：确保goroutine结束时通知WaitGroup（即使发生panic也会执行）
- wg.Wait()：阻塞主程序，直到所有worker调用Done()，计数器归零
- 如果没有wg.Wait()，主程序会立即结束，goroutine可能来不及执行
*/
```

### Mutex

Mutex 用于保护共享资源，防止多个 Goroutine 同时访问。

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

var (
    counter int
    mutex   sync.Mutex
)

func increment(name string, wg *sync.WaitGroup) {
    defer wg.Done()

    for i := 0; i < 5; i++ {
        mutex.Lock()   // 加锁
        counter++
        fmt.Printf("%s: 计数器 = %d\n", name, counter)
        mutex.Unlock() // 解锁
        time.Sleep(100 * time.Millisecond)
    }
}

func main() {
    var wg sync.WaitGroup

    // 启动两个goroutine同时修改计数器
    wg.Add(2)
    go increment("Goroutine1", &wg)
    go increment("Goroutine2", &wg)

    wg.Wait()
    fmt.Printf("最终计数器值: %d\n", counter)
}

/* 运行结果：
Goroutine1: 计数器 = 1
Goroutine2: 计数器 = 2
Goroutine1: 计数器 = 3
Goroutine2: 计数器 = 4
Goroutine1: 计数器 = 5
Goroutine2: 计数器 = 6
Goroutine1: 计数器 = 7
Goroutine2: 计数器 = 8
Goroutine1: 计数器 = 9
Goroutine2: 计数器 = 10
最终计数器值: 10

【输出结果详解】
为什么两个Goroutine会交替执行？

执行流程分析：
1. Goroutine1 先获取锁 → counter++ → 打印 → 释放锁 → 睡眠100ms
2. 在Goroutine1睡眠期间（锁已释放），Goroutine2获取锁 → counter++ → 打印 → 释放锁 → 睡眠100ms
3. 循环往复，形成交替模式

关键点：
- mutex.Lock()到mutex.Unlock()之间是临界区，同一时刻只有一个goroutine能进入
- time.Sleep()在锁外执行，这时另一个goroutine可以获取锁
- 睡眠100ms给了另一个goroutine足够的时间来执行
- 每个goroutine循环5次，所以最终counter = 2 × 5 = 10

如果没有Mutex会发生什么？
- 两个goroutine可能同时读取counter的值（比如都读到5）
- 两个goroutine都执行counter++，都写入6
- 结果：丢失了一次更新，最终值可能小于10
- Mutex确保了对共享变量counter的安全访问
*/
```

### RWMutex

RWMutex 允许多个读操作同时进行，但写操作是独占的。

````go
package main

import (
    "fmt"
    "sync"
    "time"
)

type SafeMap struct {
    mu   sync.RWMutex
    data map[string]int
}

func (sm *SafeMap) Set(key string, value int) {
    sm.mu.Lock()
    defer sm.mu.Unlock()
    sm.data[key] = value
}

func (sm *SafeMap) Get(key string) (int, bool) {
    sm.mu.RLock()
    defer sm.mu.RUnlock()
    value, ok := sm.data[key]
    return value, ok
}

func reader(sm *SafeMap, id int, wg *sync.WaitGroup) {
    defer wg.Done()
    for i := 0; i < 3; i++ {
        if value, ok := sm.Get("key"); ok {
            fmt.Printf("读取者%d: key = %d\n", id, value)
        } else {
            fmt.Printf("读取者%d: key 不存在\n", id)
        }
        time.Sleep(100 * time.Millisecond)
    }
}

func writer(sm *SafeMap, id int, wg *sync.WaitGroup) {
    defer wg.Done()
    for i := 0; i < 3; i++ {
        sm.Set("key", i*10+id)
        fmt.Printf("写入者%d: 设置 key = %d\n", id, i*10+id)
        time.Sleep(200 * time.Millisecond)
    }
}

func main() {
    sm := &SafeMap{
        data: make(map[string]int),
    }

    var wg sync.WaitGroup

    // 启动多个读取者和写入者
    for i := 1; i <= 2; i++ {
        wg.Add(1)
        go reader(sm, i, &wg)
    }

    for i := 1; i <= 2; i++ {
        wg.Add(1)
        go writer(sm, i, &wg)
    }

    wg.Wait()
}

/*
运行结果：
读取者1: key 不存在
读取者2: key 不存在
写入者1: 设置 key = 1
写入者2: 设置 key = 2
读取者1: key = 2
读取者2: key = 2
写入者1: 设置 key = 11
读取者1: key = 11
写入者2: 设置 key = 12
读取者2: key = 12

【输出结果详解】
RWMutex（读写锁）的工作原理：

读写锁特性：
- 读锁（RLock）：多个goroutine可以同时持有读锁
- 写锁（Lock）：只有一个goroutine可以持有写锁，且与读锁互斥
- 适用于读多写少的场景，提高并发性能

执行流程分析：
1. 0ms：启动2个读取者和2个写入者（4个goroutine并发运行）
2. 0ms：读取者1和2几乎同时执行Get()，都能获取RLock（读锁不互斥）
3. 0ms：虽然读操作并发执行，但fmt.Printf是顺序输出的，所以看起来是一个接一个
4. 100ms左右：读取者睡眠，写入者1获取写锁，设置key=1
5. 写入者2等待写锁（被阻塞），写入者1释放后才能获取，设置key=2
6. 读取者1和2在第二次读取时，可能同时获取RLock，都读到key=2

为什么输出看起来是顺序的，但实际是并发的？
重要理解：
- RLock允许多个goroutine同时进入临界区（真正的并发执行）
- 但fmt.Printf的输出是有先后顺序的（终端/控制台只能一行行显示）
- 虽然输出分成两行，但两个读取者实际上是几乎同时执行读取操作的

验证并发的方法：
- 两个读取者都没有被对方阻塞，可以同时进入Get()方法
- 如果用普通Mutex，第二个读取者必须等第一个释放锁才能进入
- RWMutex的读锁之间不阻塞，这就是并发！

关键知识点：
- mu.Lock()：写锁，独占访问
- mu.RLock()：读锁，共享访问
- defer mu.Unlock() / defer mu.RUnlock()：确保锁被释放
- 读锁之间不互斥，可以并发
- 写锁与任何锁都互斥
- 读写锁提高了读密集型应用的性能

RWMutex vs Mutex：
- Mutex：所有操作都互斥，简单但性能较低
- RWMutex：读读并发，读写互斥，写写互斥，性能更好

【并发执行 vs 顺序输出的区别】

关键理解：执行和输出是两回事！

并发执行的真实情况：
时间轴    读取者1                  读取者2
0ms      获取RLock ✓             获取RLock ✓  ← 同时持有锁！
1ms      读取map                 读取map      ← 真正并发！
2ms      准备打印                准备打印
3ms      fmt.Printf输出 →        等待打印
4ms      释放RLock                fmt.Printf输出
5ms                              释放RLock

输出结果：
读取者1: key 不存在  ← 先输出
读取者2: key 不存在  ← 后输出

如果用普通Mutex会怎样？
时间轴    读取者1                  读取者2
0ms      获取Lock ✓              等待Lock ✗  ← 被阻塞！
1ms      读取map                 等待...
2ms      打印输出                等待...
3ms      释放Lock                获取Lock ✓  ← 现在才能进入
4ms                              读取map
5ms                              打印输出
                                 释放Lock

总结：
- RWMutex：两个读取者几乎同时完成（并发）
- Mutex：第二个读取者必须等第一个完成（串行）
- 虽然输出都是一行行显示，但执行时间有明显差异！

【为什么读可以并发，写不能并发？】

核心原因：数据安全！

1️⃣ 读操作可以并发（多个goroutine同时读）：
原因：只读取数据，不修改数据
- 读取者A看到的值：map["key"] = 10
- 读取者B看到的值：map["key"] = 10
- 两个人同时看，互不影响！✓
- 就像多个人同时看同一本书，完全没问题

2️⃣ 写操作必须独占（一次只能一个goroutine写）：
原因：修改数据，会产生数据竞争
- 如果写入者A和B同时执行 map["key"] = 20：
  时刻1: A读取旧值10
  时刻2: B读取旧值10
  时刻3: A写入20
  时刻4: B也写入20（覆盖了A的操作）
- 更糟的情况：map内部结构被破坏，程序崩溃！✗

3️⃣ 读写也不能并发（读的时候不能写）：
原因：读到一半被修改，数据不一致
- 读取者正在读：map["key"] = 10
- 写入者突然修改：map["key"] = 20
- 读取者可能读到一半旧值，一半新值（数据撕裂）
- 结果不可预测！✗

【形象比喻】

图书馆规则：
✅ 多人同时阅读同一本书 → 允许（读-读并发）
✗ 有人在写书时，别人不能看 → 禁止（写-读互斥）
✗ 不能多人同时在一本书上写字 → 禁止（写-写互斥）

【技术细节】

安全的场景（允许并发）：
```go
// 多个goroutine同时读
goroutine1: value := map["key"]  // 读取
goroutine2: value := map["key"]  // 读取
// 结果：都安全，都读到相同的值
````

不安全的场景（必须互斥）：

```go
// 多个goroutine同时写
goroutine1: map["key"] = 100     // 写入
goroutine2: map["key"] = 200     // 写入
// 问题：最终值是100还是200？不确定！数据竞争！

// 读写同时发生
goroutine1: value := map["key"]  // 读取
goroutine2: map["key"] = 100     // 写入
// 问题：读取者可能读到脏数据或崩溃！
```

【RWMutex 的智能之处】

RWMutex 根据操作类型自动选择策略：

- RLock()：允许多个并发（读操作安全）
- Lock()：强制独占（写操作危险）

性能对比：
场景：10 个读，1 个写

- 普通 Mutex：11 个操作全部串行，耗时 11x
- RWMutex：10 个读并发，1 个写独占，耗时 ≈ 2x
- 性能提升：5.5 倍！

本质：
读写锁是一种优化，在保证数据安全的前提下，
尽可能允许并发，提高性能。

package main

import (
    "fmt"
    "math/rand"
    "sync"
    "time"
)

// 生产者
func producer(id int, jobs chan<- int, wg *sync.WaitGroup) {
    defer wg.Done()

    for i := 0; i < 5; i++ {
        job := rand.Intn(100)
        jobs <- job
        fmt.Printf("生产者%d 生产了任务: %d\n", id, job)
        time.Sleep(time.Duration(rand.Intn(1000)) * time.Millisecond)
    }
}

// 消费者
func consumer(id int, jobs <-chan int, results chan<- int, wg *sync.WaitGroup) {
    defer wg.Done()

    for job := range jobs {
        // 模拟处理任务
        time.Sleep(time.Duration(rand.Intn(1500)) * time.Millisecond)
        result := job * 2
        results <- result
        fmt.Printf("消费者%d 处理了任务 %d, 结果: %d\n", id, job, result)
    }
}

func main() {
    rand.Seed(time.Now().UnixNano())

    // 创建channel
    jobs := make(chan int, 10)
    results := make(chan int, 10)

    var wg1, wg2 sync.WaitGroup

    // 启动生产者
    for i := 1; i <= 2; i++ {
        wg1.Add(1)
        go producer(i, jobs, &wg1)
    }

    // 启动消费者
    for i := 1; i <= 3; i++ {
        wg2.Add(1)
        go consumer(i, jobs, results, &wg2)
    }

    // 等待所有生产者完成
    wg1.Wait()
    close(jobs) // 关闭jobs channel

    // 等待所有消费者完成
    wg2.Wait()
    close(results) // 关闭results channel

    // 收集所有结果
    fmt.Println("\n=== 最终结果 ===")
    for result := range results {
        fmt.Printf("结果: %d\n", result)
    }

    fmt.Println("所有任务处理完成")
}

/*
【代码详解】生产者-消费者模型

这是一个经典的并发设计模式，展示了如何使用Channel和WaitGroup协调多个goroutine。

核心组件：
1. jobs channel：生产者向其发送任务（容量10）
2. results channel：消费者向其发送处理结果（容量10）
3. wg1：等待所有生产者完成
4. wg2：等待所有消费者完成

执行流程：
1. 启动2个生产者goroutine：
   - 每个生产5个随机任务（0-99）
   - 发送到jobs channel
   - 随机延迟0-1000ms模拟生产时间

2. 启动3个消费者goroutine：
   - 从jobs channel接收任务
   - 处理任务（×2）并延迟0-1500ms
   - 将结果发送到results channel
   - 使用for job := range jobs自动接收直到channel关闭

3. 主流程控制：
   - wg1.Wait()：等待所有生产者完成
   - close(jobs)：关闭jobs channel，通知消费者没有新任务了
   - wg2.Wait()：等待所有消费者处理完现有任务
   - close(results)：关闭results channel
   - 从results收集所有结果并打印

关键设计点：
- 带缓冲channel避免生产者和消费者频繁阻塞
- chan<- 和 <-chan 类型约束，防止误用
- defer wg.Done()确保计数器正确递减
- 关闭channel的时机很重要：生产者完成后才关闭jobs
- range自动处理channel关闭，优雅退出循环
- 生产者数量和消费者数量可以不同（2个生产者，3个消费者）

应用场景：
- 任务队列处理
- 数据流水线
- Web请求处理
- 批量数据处理
*/

// 练习题1: 并发计算斐波那契数列
func fibonacci(n int, ch chan<- int, wg *sync.WaitGroup) {
    defer wg.Done()

    if n <= 1 {
        ch <- n
        return
    }

    a, b := 0, 1
    for i := 2; i <= n; i++ {
        a, b = b, a+b
    }
    ch <- b
}

func calculateFibonacciConcurrent(n int) {
    ch := make(chan int, n)
    var wg sync.WaitGroup

    start := time.Now()

    for i := 0; i < n; i++ {
        wg.Add(1)
        go fibonacci(i, ch, &wg)
    }

    // 启动一个goroutine来关闭channel
    go func() {
        wg.Wait()
        close(ch)
    }()

    // 收集结果
    results := make([]int, n)
    i := 0
    for result := range ch {
        results[i] = result
        i++
    }

    elapsed := time.Since(start)
    fmt.Printf("并发计算斐波那契数列前%d项: %v\n", n, results)
    fmt.Printf("耗时: %v\n", elapsed)
}

// 练习题2: 简单的并发Web爬虫
type Crawler struct {
    visited map[string]bool
    mu      sync.Mutex
}

func (c *Crawler) Crawl(url string, depth int, wg *sync.WaitGroup) {
    defer wg.Done()

    if depth <= 0 {
        return
    }

    // 检查是否已访问
    c.mu.Lock()
    if c.visited[url] {
        c.mu.Unlock()
        return
    }
    c.visited[url] = true
    c.mu.Unlock()

    // 模拟爬取网页
    fmt.Printf("爬取URL: %s (深度: %d)\n", url, depth)
    time.Sleep(time.Duration(rand.Intn(500)) * time.Millisecond)

    // 模拟获取链接
    links := []string{
        url + "/page1",
        url + "/page2",
        url + "/page3",
    }

    // 并发爬取链接
    var subWg sync.WaitGroup
    for _, link := range links {
        subWg.Add(1)
        go c.Crawl(link, depth-1, &subWg)
    }
    subWg.Wait()
}

func crawlWebsites() {
    crawler := &Crawler{
        visited: make(map[string]bool),
    }

    urls := []string{
        "http://example.com",
        "http://golang.org",
        "http://github.com",
    }

    var wg sync.WaitGroup
    for _, url := range urls {
        wg.Add(1)
        go crawler.Crawl(url, 2, &wg)
    }

    wg.Wait()
    fmt.Printf("总共访问了 %d 个URL\n", len(crawler.visited))
}

// 练习题3: 生产者-消费者处理数据
type DataProcessor struct {
    data     []int
    processed []int
    mu       sync.Mutex
}

func (dp *DataProcessor) Producer(dataChan chan<- int, wg *sync.WaitGroup) {
    defer wg.Done()

    for _, item := range dp.data {
        dataChan <- item
        fmt.Printf("生产数据: %d\n", item)
        time.Sleep(100 * time.Millisecond)
    }
}

func (dp *DataProcessor) Consumer(dataChan <-chan int, resultChan chan<- int, wg *sync.WaitGroup) {
    defer wg.Done()

    for data := range dataChan {
        // 模拟数据处理
        processed := data * data
        resultChan <- processed
        fmt.Printf("处理数据: %d -> %d\n", data, processed)
        time.Sleep(150 * time.Millisecond)
    }
}

func (dp *DataProcessor) Collector(resultChan <-chan int, wg *sync.WaitGroup) {
    defer wg.Done()

    for result := range resultChan {
        dp.mu.Lock()
        dp.processed = append(dp.processed, result)
        dp.mu.Unlock()
    }
}

func processData() {
    processor := &DataProcessor{
        data: make([]int, 0),
    }

    // 生成测试数据
    for i := 1; i <= 10; i++ {
        processor.data = append(processor.data, i)
    }

    dataChan := make(chan int, 5)
    resultChan := make(chan int, 5)

    var wg sync.WaitGroup

    // 启动生产者
    wg.Add(1)
    go processor.Producer(dataChan, &wg)

    // 启动消费者
    for i := 0; i < 3; i++ {
        wg.Add(1)
        go processor.Consumer(dataChan, resultChan, &wg)
    }

    // 启动收集器
    wg.Add(1)
    go processor.Collector(resultChan, &wg)

    // 等待生产者完成并关闭channel
    go func() {
        wg.Wait()
        close(dataChan)
    }()

    // 等待所有goroutine完成并关闭resultChan
    go func() {
        wg.Wait()
        close(resultChan)
    }()

    // 等待所有任务完成
    wg.Wait()

    fmt.Printf("原始数据: %v\n", processor.data)
    fmt.Printf("处理后数据: %v\n", processor.processed)
}
