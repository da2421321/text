# 04 - 三层架构实战演示 (Controller-Service-Repository)

这节课我们将前几课所有的知识融会贯通，演示如何按照大型架构标准的“三层架构”写一个实战级的**用户注册接口**。

这三层分别是：
1. **Controller**: 对外接待，负责拆解请求 (Context 解析 JSON)、初步参数合法性验证，然后分配给下面的人干活，最后包裹响应返回给前端。
2. **Service**: 业务核心，只关注于业务规则。比如判断“该用户名是否被注册过”、“密码是否强度够”、“给密码进行哈希加盐加密”。
3. **Repository(DAO)**: 底层苦力，跟数据库对话，接收拼装好的模型，丢给 GORM 去持久化。

我们假设全局只有一个实例 `db *gorm.DB`。真正的企业代码可能会用到**依赖注入 (Dependency Injection)**，为了便于前端思维转过来，我们这里演示简化的平铺式依赖传递。

## 第一层：Repository (数据访问层)
在 `internal/repository/user.go` 中：

```go
package repository

import (
	"my-go-backend/internal/model"
	"gorm.io/gorm"
)

type UserRepository struct {
	DB *gorm.DB // 持有数据库连接池
}

func (r *UserRepository) FindByUsername(username string) (*model.User, error) {
	var user model.User
	// 使用 GORM 构建 SQL
	err := r.DB.Where("username = ?", username).First(&user).Error
	if err != nil && err != gorm.ErrRecordNotFound {
		return nil, err
	}
	if err == gorm.ErrRecordNotFound {
		return nil, nil // 没找到不算是真正的系统报错，告诉上层是 nil
	}
	return &user, nil
}

func (r *UserRepository) CreateUser(user *model.User) error {
	return r.DB.Create(user).Error
}
```

## 第二层：Service (业务层)
在 `internal/service/user.go` 中，这里**绝不允许导入任何特定 Web 框架(Gin)的包**，保持纯粹！

```go
package service

import (
	"errors"
	"my-go-backend/internal/model"
	"my-go-backend/internal/repository"
	// 假设我们有个工具包用 bcrypt 做了加密封装
	"my-go-backend/internal/pkg/crypto" 
)

type UserService struct {
	Repo *repository.UserRepository // 依赖注入 Repository 层
}

// 供 Controller 调用的执行主入口
func (s *UserService) Register(username, password string) error {
	// 核心业务规则 1: 检查是否已被注册
	existingUser, err := s.Repo.FindByUsername(username)
	if err != nil {
		return err // 数据库挂了
	}
	if existingUser != nil {
		return errors.New("用户名已被注册") // 自定义业务校验拒绝
	}

	// 核心业务规则 2: 处理密码存储安全
	hashedPassword, err := crypto.HashPassword(password)
	if err != nil {
		return err
	}

	// 新建要插入数据库的对象
	newUser := &model.User{
		Username: username,
		Password: hashedPassword,
	}

	// 派发给 Repo 去干底层的脏活
	return s.Repo.CreateUser(newUser)
}
```

## 第三层：Controller (应用控制器层)
在 `internal/controller/user.go` 中：

```go
package controller

import (
	"github.com/gin-gonic/gin"
	"my-go-backend/internal/service"
	"net/http"
)

type UserController struct {
	Svc *service.UserService // 依赖注入 Service 层
}

// 定义用于接收前端入参的 DTO
type RegisterReq struct {
	Username string `json:"username" binding:"required,min=4,max=16"` // Validation 验证器 tag
	Password string `json:"password" binding:"required,min=6"`
}

func (ctrl *UserController) HandleRegister(c *gin.Context) {
	var req RegisterReq
	// 1. Controller 接客并校验参数
	if err := c.ShouldBindJSON(&req); err != nil {
		// 校验不通过直接打回
		c.JSON(http.StatusBadRequest, gin.H{"code": 400, "msg": "参数格式不正确: " + err.Error()})
		return
	}

	// 2. 将摘出来的合法参数交给 Service 处理核心业务规则
	err := ctrl.Svc.Register(req.Username, req.Password)
	if err != nil {
		// 业务层返回的错误，我们包装成 200 HTTP状态码（国内业务习惯），但内部 code 为失败 500
		c.JSON(http.StatusOK, gin.H{"code": 500, "msg": err.Error()})
		return
	}

	// 3. 一切顺利，包装格式化响应给前端
	c.JSON(http.StatusOK, gin.H{"code": 200, "msg": "注册成功"})
}
```

## 串联：组装入口 `main.go`
```go
func main() {
    db := InitGorm() // 假设这能连上数据库获取单例指针

    // 像拼积木一样把它们依赖组装起来
    userRepo := &repository.UserRepository{DB: db}
    userService := &service.UserService{Repo: userRepo}
    userCtrl := &controller.UserController{Svc: userService}

    r := gin.Default()
    r.POST("/register", userCtrl.HandleRegister)
    
    r.Run(":8080")
}
```
经过这一套划分，不管以后项目加多少功能，每个文件只负责自己该做的事，互不干扰，清晰明了！

## 生产补充 1：业务错误与系统错误分离
建议 Service 返回“可识别业务错误”，Controller 再映射成统一错误码：

```go
var ErrUserAlreadyExists = errors.New("user already exists")

func (s *UserService) Register(username, password string) error {
	existingUser, err := s.Repo.FindByUsername(username)
	if err != nil {
		return err
	}
	if existingUser != nil {
		return ErrUserAlreadyExists
	}
	// ...
	return nil
}
```

Controller 中：

```go
if err := ctrl.Svc.Register(req.Username, req.Password); err != nil {
    if errors.Is(err, service.ErrUserAlreadyExists) {
        response.Error(c, 40001, "用户名已存在")
        return
    }
    response.Error(c, 50000, "系统繁忙")
    return
}
```

## 生产补充 2：跨表写操作放在 Service 事务里
若注册流程包含“创建用户 + 初始化钱包 + 写审计日志”，应在 Service 中统一事务，避免部分成功：

```go
err := db.Transaction(func(tx *gorm.DB) error {
    // 使用 tx 贯穿 repo
    return nil
})
```

事务边界放在 Service，而不是 Controller 或单个 Repository。
