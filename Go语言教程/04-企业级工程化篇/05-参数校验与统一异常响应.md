# 05 - 参数校验与标准统一响应

作为前后端对接的桥梁，不严谨的 API 往往会引发大量 Bug。有两个企业级实践：
1. **统一返回数据结构 (Response Formatter)**：任何成功的请求和失败报错的结构体长得必须一模一样，不然纯折腾前端转译和全局拦截器。
2. **Validator 参数校验机制**：借助 `binding` 和结构体 tag (标签) 将请求体校验全部自动化。

## 1. 定义统一的 JSON 返回封装结构

不要在每一个 Controller 里去散装拼 `gin.H{"code": xxx, "msg": xxx, "data": xxx}`。
写一个公用的包 `internal/pkg/response/response.go`：

```go
package response

import (
	"github.com/gin-gonic/gin"
	"net/http"
)

// ResponseData 标准返回结构体
type ResponseData struct {
	Code int         `json:"code"`
	Msg  string      `json:"msg"`
	Data interface{} `json:"data,omitempty"` // omitempty：如果是nil或空值，就不序列化回传这把钥匙
}

const (
	SuccessCode = 200
	ErrorCode   = 500
)

// 响应成功
func Success(c *gin.Context, data interface{}) {
	c.JSON(http.StatusOK, ResponseData{
		Code: SuccessCode,
		Msg:  "success",
		Data: data,
	})
}

// 响应失败 (向前端抛出具体的业务错误信息)
func Error(c *gin.Context, code int, msg string) {
	c.JSON(http.StatusOK, ResponseData{
		Code: code,
		Msg:  msg,
		Data: nil,
	})
}
```
用法很简单，在 Controller 里直接调用 `response.Success(c, userObj)`，前端就会永远收到 `{ code: 200, msg: "success", data: {...} }` 这样工工整整的数据包。

## 2. Validator 实战：利用 Struct Tag

Gin 框架内置了 [go-playground/validator](https://github.com/go-playground/validator) 这个校验利器。无需手写大量的 `if req.Age > 18` 这类逻辑代码，直接在 DTO 里下指令：

```go
type CreateUserReq struct {
    // binding:"required" 意味着这个字段如果是空就会自动报错打回
	Username string `json:"username" binding:"required,min=4,max=16"`
	
	// 可以要求必须是邮箱格式
	Email string `json:"email" binding:"required,email"`
	
	// 要求数值范围
	Age int `json:"age" binding:"gte=18,lte=130"` // 大于等于18且小于等于130
	
	// 在 Enum 类型的几个值里选（常用于状态枚举）
	Role string `json:"role" binding:"required,oneof=admin worker user"`
}

func HandleCreate(c *gin.Context) {
    var req CreateUserReq
    // 只有所有 binding 规则都通过，ShouldBindJSON 才不会报错
    if err := c.ShouldBindJSON(&req); err != nil {
        response.Error(c, 400, "参数未通过校验: " + err.Error())
        return
    }
    
    // 参数不仅合法干净被校验过，并且已经绑定到结构体，安全进入 Service 执行业务
    // ...
}
```

将这五个核心工程化能力（**三层架构、环境配置文件、鉴权中间件、数据库DAO、统一响应与参数护盾**）结合起来，你所写的程序就立刻从“玩具 Demo” 蜕变为了“企业规范级微服务”。这套模式在各个大型互联网后端公司通用！

## 3. 错误码建议分段管理

避免“全是 500”这种不可观测设计，建议分段：

- `0`：成功
- `4xxxx`：业务错误（如参数非法、用户不存在、库存不足）
- `5xxxx`：系统错误（数据库异常、依赖服务故障）

这样前端、日志系统、告警平台都能快速定位问题类别。

## 4. 自定义校验示例（手机号）
内置规则不够时可注册自定义验证器：

```go
validate := binding.Validator.Engine().(*validator.Validate)
_ = validate.RegisterValidation("mobile", func(fl validator.FieldLevel) bool {
    v := fl.Field().String()
    re := regexp.MustCompile(`^1[3-9]\d{9}$`)
    return re.MatchString(v)
})

type Req struct {
    Mobile string `json:"mobile" binding:"required,mobile"`
}
```

## 5. Controller 不要直接透传底层错误
`err.Error()` 可能暴露 SQL、表名、内部路径。建议：

- 返回用户可读的通用提示。
- 详细错误写日志（带 requestID）。
