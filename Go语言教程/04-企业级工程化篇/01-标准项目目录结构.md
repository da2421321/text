# 01 - 标准企业级业务目录结构 (Standard Go Project Layout)

在 Node.js (NestJS 等) 世界里，我们习惯了模块化与依赖注入（Controller, Service, Repository）。在 Go 当中，虽然官方没有强制要求，但业界约定俗成了一套基于[标准 Go 项目结构 (Standard Go Project Layout)](https://github.com/golang-standards/project-layout) 与 DDD (领域驱动设计) 结合的目录划分方式。

## 经典三层架构目录 (适合写 API 服务)

创建一个真正的线上项目时，你的目录应该长这样：

```text
my-go-backend/
├── cmd/                # 程序的入口点
│   └── server/
│       └── main.go     # main 函数所在，只做初始化和启动服务器的操作
├── internal/           # 核心私有业务逻辑（Go 编译器会阻止外部项目导入 internal 下的代码）
│   ├── config/         # 配置文件读取和映射 (Viper)
│   ├── controller/     # 控制器层：负责解析 HTTP 请求 (Gin Context)，格式化返回参数
│   ├── service/        # 服务层：核心业务逻辑运算，无视框架，只写纯粹的判断运算
│   ├── repository/     # 数据访问层 (DAO)：所有查读写数据库 (GORM) 的代码放这里
│   ├── model/          # 所有的数据库表结构 (Struct) 与各种接口返回体 (DTO)
│   ├── middleware/     # 中间件（鉴权 JWT、跨域 CORS、限流等）
│   └── pkg/            # 放置所有通用的、与业务不耦合的工具包（如加密、请求、时间格式化）
├── config/             # YAML/JSON 配置文件物理存放处 (例如 config.yaml)
├── logs/               # 运行时产生的日志文件
├── go.mod              # 依赖管理，相当于 package.json
└── README.md
```

## 各层职责划分与流转图

**请求进来的流转顺序：**
`Router (路由分配)` -> `Controller (参数校验)` -> `Service (执行业务规则)` -> `Repository (操作数据库)`
**返回回去的流转顺序相反。**

1. **Controller 层不能写数据库操作和复杂逻辑**，它只干三件事：“拿参数 -> 叫 Service 干活 -> 把结果变 JSON 丢给客户端”。
2. **Service 层不能出现 `*gin.Context`**，它必须与特定的 Web 框架解耦（意味着哪怕以后把 Gin 换成别的，Service 也不用改一行代码）。
3. **Repository 层只负责一条**：拼 SQL / 调 GORM。不要在这里做“如果不满足 X 条件就报错”的业务验证。

这是大型项目保证可扩展性和好测试（Mock）的第一性原理。
