> **本章目标**：理解 Java 异常体系，知道错误该在哪里处理；掌握泛型的核心写法，写出类型安全、可复用的后端代码

> **预计耗时**：5~8 小时  
> **前置要求**：完成基础篇，至少理解类、接口、集合、方法、面向对象

## 一、为什么这章对前端转全栈很关键？

如果你是前端同学，看到“异常处理”和“泛型”，第一反应可能是：

- 异常不就是 `try/catch` 吗？
- 泛型不就是 TypeScript 里的 `<T>` 吗？

方向没错，但 **Java 后端里的重要程度比前端高很多**。

原因很简单：

1. 后端代码不是点一下按钮就结束，而是要处理接口请求、数据库、缓存、文件、消息队列这些不稳定因素
2. 一个异常如果处理不好，可能不是页面报错，而是接口 500、事务回滚、线程中断、日志爆炸
3. 泛型在 Java 后端里几乎无处不在，`List<User>`、`Result<T>`、`PageResult<T>`、`Optional<T>` 都是日常写法

你可以先建立一个对应关系：

| 前端里的概念 | Java 后端里的对应物 |
|------|------|
| `try/catch` | `try-catch-finally` |
| `throw new Error("xxx")` | `throw new RuntimeException("xxx")` / `throw new BizException(...)` |
| `Promise.reject(err)` | 方法抛异常，交给上层处理 |
| `axios.interceptors.response.use(..., err => ...)` | 全局异常处理器 |
| `type ApiResult<T> = { code; message; data }` | `class Result<T>` |
| `Array<User>` / `Map<string, User>` | `List<User>` / `Map<String, User>` |

一句话总结：

**异常处理解决“程序出错时怎么办”，泛型解决“同一套代码如何安全地服务多种类型”。**

## 二、异常到底是什么？

### 2.1 白话解释

异常就是：**程序在运行过程中，出现了“不符合预期”的情况，导致当前流程无法正常继续**。

比如一个后端接口：

```text
浏览器发请求
  -> Controller 接收参数
  -> Service 处理业务
  -> Repository 查数据库
  -> 返回 JSON
```

这个过程中每一步都可能出错：

- 参数为空
- 用户不存在
- 库存不足
- 数据库连接失败
- 文件读取失败
- 空指针

这些“出错”在 Java 里通常就表现为异常对象。

### 2.2 一个前端更容易理解的类比

你可以把异常理解成前端里的这些情况：

- `JSON.parse("abc")` 失败
- `undefined.name` 报错
- `fetch()` 网络失败
- 路由参数不合法
- 表单校验没通过

区别在于：

- 前端报错，通常影响一个页面或一个交互
- 后端报错，可能直接影响一个接口、一个事务、甚至一批请求

所以后端比前端更强调：

- 哪些错误要拦截
- 哪些错误要继续往上抛
- 最终返回给前端什么信息
- 日志里保留什么信息给开发排查

### 2.3 没有异常处理会怎样？

```java
public class NoExceptionHandling {
    public static void main(String[] args) {
        int[] arr = {1, 2, 3};
        System.out.println(arr[10]); // 数组越界
        System.out.println("这行不会执行");
    }
}
```

运行到 `arr[10]` 时，JVM 会直接抛出异常，中断当前流程。

如果写了异常处理：

```java
public class WithExceptionHandling {
    public static void main(String[] args) {
        int[] arr = {1, 2, 3};

        try {
            System.out.println(arr[10]);
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("数组下标越界，最大下标是: " + (arr.length - 1));
        }

        System.out.println("程序继续执行");
    }
}
```

这时候程序不会直接崩掉，而是进入 `catch`，你就有机会决定怎么处理。

### 2.4 先学会看异常栈

很多前端同学刚接触 Java 时，会被一长串报错吓到。其实看法很固定。

例如：

```text
Exception in thread "main" java.lang.NumberFormatException: For input string: "abc"
    at java.base/java.lang.Integer.parseInt(Integer.java:668)
    at demo.UserService.parseAge(UserService.java:12)
    at demo.UserService.main(UserService.java:6)
```

你只要按这个顺序看：

1. **异常类型**：`NumberFormatException`
2. **异常信息**：`For input string: "abc"`
3. **第一行你自己的业务代码**：`UserService.java:12`
4. **调用链**：谁调了谁，最后一路传到了 `main`

排错时最重要的是：

- 先看第一行异常类型
- 再看第一处你自己的代码位置
- 不要先盯着 JDK 内部代码

## 三、Java 异常体系全景图

Java 的异常顶层父类是 `Throwable`：

```text
Throwable
├── Error
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   └── ...
└── Exception
    ├── RuntimeException
    │   ├── NullPointerException
    │   ├── IllegalArgumentException
    │   ├── NumberFormatException
    │   ├── IndexOutOfBoundsException
    │   └── ...
    └── 其他 Exception
        ├── IOException
        ├── SQLException
        ├── ClassNotFoundException
        └── ...
```

### 3.1 Error 和 Exception 的区别

| 类型 | 含义 | 你通常怎么做 |
|------|------|------|
| `Error` | JVM 层面的严重问题 | 一般不处理，先排查环境或代码设计 |
| `Exception` | 程序运行中的异常情况 | 这是你重点学习和处理的对象 |

常见理解：

- `OutOfMemoryError`：内存不够，通常不是 `try/catch` 能优雅解决的
- `StackOverflowError`：大概率是递归写错了
- `Exception`：业务错误、参数错误、IO 错误、数据库错误等

### 3.2 受检异常和非受检异常

这是 Java 初学者必须过的一关。

| 类型 | 受检异常 Checked Exception | 非受检异常 Unchecked Exception |
|------|------|------|
| 继承关系 | `Exception`，但不是 `RuntimeException` | `RuntimeException` 及其子类 |
| 编译器是否强制处理 | 是 | 否 |
| 常见场景 | 文件、网络、数据库、反射等外部操作 | 参数错误、空指针、状态错误、业务异常 |
| 常见例子 | `IOException`、`SQLException` | `NullPointerException`、`IllegalArgumentException` |

### 3.3 你该怎么记？

可以用一句人话区分：

- **受检异常**：这件事天然就可能失败，你必须提前考虑
- **非受检异常**：这通常是代码逻辑有问题，或者业务条件不满足

再翻译成前端语言：

- `IOException` 像“你去请求一个外部资源，但这个资源本身可能失败”
- `IllegalArgumentException` 像“你自己把参数传错了”

### 3.4 在真实项目里怎么选？

现代 Java 后端，尤其是 Spring Boot 项目里，有一个常见实践：

- **业务异常** 通常继承 `RuntimeException`
- **底层基础设施异常** 由框架或底层库抛出
- 在边界处统一转换成标准响应

为什么很多业务异常不做成受检异常？

因为受检异常会让每一层方法签名都写 `throws`，层层传递后会很啰嗦。业务异常更常见的做法是直接继承 `RuntimeException`，然后交给全局异常处理器统一处理。

## 四、异常是如何传播的：`throw`、`throws`、调用栈回退

### 4.1 `throw` 和 `throws` 的区别

很多人会混。

| 写法 | 含义 |
|------|------|
| `throw` | 真正“抛出一个异常对象” |
| `throws` | 在方法签名上声明“这个方法可能抛出某种异常” |

示例：

```java
public class ThrowDemo {

    public static void checkAge(int age) {
        if (age < 0) {
            throw new IllegalArgumentException("年龄不能小于 0");
        }
    }
}
```

这里 `throw` 的意思是：马上抛异常，中断当前方法。

再看 `throws`：

```java
import java.io.IOException;
import java.nio.file.Files;
import java.nio.file.Path;

public class ThrowsDemo {

    public static String readConfig(String path) throws IOException {
        return Files.readString(Path.of(path));
    }
}
```

这里 `throws IOException` 的意思是：

“我这个方法在读文件，可能失败；至于怎么处理，交给调用我的人决定。”

### 4.2 异常传播的本质

当一个方法内部抛出异常后，如果当前方法没有处理，这个异常就会沿着调用链一直往上冒。

看一个完整示例：

```java
class BizException extends RuntimeException {
    public BizException(String message) {
        super(message);
    }
}

public class ExceptionFlowDemo {

    public static void main(String[] args) {
        try {
            controller();
        } catch (BizException e) {
            System.out.println("main 捕获到异常: " + e.getMessage());
        }
    }

    static void controller() {
        service();
    }

    static void service() {
        repository();
    }

    static void repository() {
        throw new BizException("用户不存在");
    }
}
```

执行路径：

```text
repository 抛异常
  -> service 没处理，继续往上抛
  -> controller 没处理，继续往上抛
  -> main catch 到
```

这个过程就叫：**调用栈回退**。

### 4.3 在后端分层里，应该在哪里处理异常？

这是前端转后端最容易困惑的点之一。

先看常见分层：

```text
Controller -> Service -> Repository
```

一般建议：

| 层级 | 更适合做什么 |
|------|------|
| `Controller` | 接收请求、返回响应，尽量少写复杂异常处理 |
| `Service` | 处理业务规则，适合抛业务异常 |
| `Repository` / DAO | 负责数据库访问，通常不关心前端返回文案 |
| 全局异常处理器 | 最终统一把异常转成 JSON 响应 |

换句话说：

- **能处理并恢复流程**，就地处理
- **不能处理，只是知道出错了**，继续往上抛

### 4.4 一个更真实的后端思路

```java
class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(Long userId) {
        super("用户不存在: " + userId);
    }
}

class UserRepository {
    public String findNameById(Long userId) {
        if (userId.equals(999L)) {
            return null;
        }
        return "张三";
    }
}

class UserService {
    private final UserRepository userRepository = new UserRepository();

    public String getUserName(Long userId) {
        String name = userRepository.findNameById(userId);
        if (name == null) {
            throw new UserNotFoundException(userId);
        }
        return name;
    }
}

public class LayerDemo {
    public static void main(String[] args) {
        UserService userService = new UserService();

        try {
            String name = userService.getUserName(999L);
            System.out.println(name);
        } catch (UserNotFoundException e) {
            System.out.println("给前端返回 404 风格的业务提示: " + e.getMessage());
        }
    }
}
```

这里的关键不是“能不能 catch”，而是：

- Repository 只负责取数据
- Service 决定“查不到用户”是不是业务错误
- 更上层再决定返回 404、400 还是统一错误响应

## 五、`try-catch-finally` 详解

### 5.1 基本结构

```java
try {
    // 可能抛异常的代码
} catch (ExceptionType1 e) {
    // 处理异常 1
} catch (ExceptionType2 e) {
    // 处理异常 2
} finally {
    // 无论是否异常，通常都会执行
}
```

### 5.2 执行顺序

你可以这样理解：

1. 先执行 `try`
2. 如果没异常，跳过所有 `catch`
3. 如果发生异常，匹配最合适的 `catch`
4. 最后执行 `finally`

### 5.3 完整示例

```java
import java.util.Scanner;

public class TryCatchDemo {
    public static void main(String[] args) {
        Scanner scanner = new Scanner(System.in);

        try {
            System.out.print("请输入一个整数: ");
            String input = scanner.nextLine();

            int number = Integer.parseInt(input); // 用户可能输入 abc
            int result = 100 / number;            // 用户可能输入 0

            System.out.println("结果: " + result);
        } catch (NumberFormatException e) {
            System.out.println("输入不是有效整数");
        } catch (ArithmeticException e) {
            System.out.println("除数不能为 0");
        } finally {
            System.out.println("程序结束前执行 finally");
            scanner.close();
        }
    }
}
```

### 5.4 `catch` 的顺序很重要

子类异常要写在前面，父类异常写在后面。

错误示例：

```java
try {
    // ...
} catch (Exception e) {
    // 太宽了
} catch (NumberFormatException e) {
    // 永远到不了这里
}
```

因为 `NumberFormatException` 本来就是 `Exception` 的子类，前面已经被拦住了。

正确写法：

```java
try {
    // ...
} catch (NumberFormatException e) {
    // 更具体
} catch (Exception e) {
    // 最后兜底
}
```

### 5.5 多异常合并捕获

如果处理逻辑一样，可以合并：

```java
try {
    // ...
} catch (NumberFormatException | ArithmeticException e) {
    System.out.println("输入或计算错误: " + e.getMessage());
}
```

### 5.6 `finally` 常见注意点

`finally` 常用于：

- 关闭文件
- 关闭网络连接
- 关闭数据库连接
- 做一些收尾动作

注意：

1. 不要在 `finally` 里写复杂业务逻辑
2. 不要在 `finally` 里 `return`
3. 不要在 `finally` 里再抛出新的异常把原异常覆盖掉

## 六、`try-with-resources`：自动关闭资源

### 6.1 为什么它比 `finally` 更常用？

文件流、网络流、数据库连接这类资源，用完必须关。

传统写法：

```java
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;

public class OldStyleDemo {
    public static void main(String[] args) {
        BufferedReader reader = null;

        try {
            reader = new BufferedReader(new FileReader("data.txt"));
            System.out.println(reader.readLine());
        } catch (IOException e) {
            e.printStackTrace();
        } finally {
            if (reader != null) {
                try {
                    reader.close();
                } catch (IOException e) {
                    e.printStackTrace();
                }
            }
        }
    }
}
```

问题很明显：

- 代码长
- 容易忘记关
- 关闭本身也可能出错

### 6.2 推荐写法

```java
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;

public class TryWithResourcesDemo {
    public static void main(String[] args) {
        try (BufferedReader reader = new BufferedReader(new FileReader("data.txt"))) {
            System.out.println(reader.readLine());
        } catch (IOException e) {
            System.out.println("读取文件失败: " + e.getMessage());
        }
    }
}
```

只要资源实现了 `AutoCloseable` 接口，就能写进 `try (...)` 里，Java 会自动调用 `close()`。

### 6.3 多个资源一起管理

```java
import java.io.BufferedReader;
import java.io.BufferedWriter;
import java.io.FileReader;
import java.io.FileWriter;
import java.io.IOException;

public class MultiResourceDemo {
    public static void main(String[] args) {
        try (
            BufferedReader reader = new BufferedReader(new FileReader("input.txt"));
            BufferedWriter writer = new BufferedWriter(new FileWriter("output.txt"))
        ) {
            String line;
            while ((line = reader.readLine()) != null) {
                writer.write(line);
                writer.newLine();
            }
        } catch (IOException e) {
            System.out.println("文件复制失败: " + e.getMessage());
        }
    }
}
```

### 6.4 在后端开发里的典型场景

- 读写文件
- 处理 Excel、CSV、PDF
- 手动操作输入输出流
- 某些三方 SDK 返回的可关闭对象

补充一点：

在 Spring Boot 里，数据库连接通常交给连接池和框架管理，不是你手动 `close()` 每个连接；但只要你自己拿到了“资源对象”，就要有关闭意识。

## 七、自定义异常：把业务规则表达清楚

### 7.1 为什么不能总用 `RuntimeException`？

当然可以直接写：

```java
throw new RuntimeException("库存不足");
```

但这会有几个问题：

1. 信息太泛
2. 不方便分类处理
3. 不方便统一错误码
4. 日志和前端返回不够规范

所以真实项目里通常会定义自己的业务异常。

### 7.2 一个更接近真实项目的写法

在真实项目里,我们通常会分三步来设计业务异常体系:

1. **定义错误码枚举** - 统一管理所有业务错误类型
2. **定义业务异常类** - 封装错误码和错误信息
3. **在业务代码里使用** - 在合适的地方抛出异常

#### 第一步:定义错误码枚举

```java
/**
 * 错误码枚举
 * 
 * 设计思路:
 * - code: 错误码,用于前后端约定,方便前端做不同处理
 * - message: 错误提示,直接返回给前端展示
 * 
 * 错误码规范(参考 HTTP 状态码风格):
 * - 40000~49999: 客户端错误(参数错误、业务规则不满足等)
 * - 50000~59999: 服务端错误(系统异常、数据库异常等)
 */
public enum ErrorCode {
    // 参数相关错误 (400xx)
    INVALID_PARAM(40000, "参数不合法"),
    
    // 资源不存在错误 (404xx)
    USER_NOT_FOUND(40400, "用户不存在"),
    ORDER_NOT_FOUND(40401, "订单不存在"),
    PRODUCT_NOT_FOUND(40402, "商品不存在"),
    
    // 业务规则错误 (400xx)
    INSUFFICIENT_BALANCE(40001, "余额不足"),
    OUT_OF_STOCK(40002, "库存不足"),
    ORDER_STATUS_ERROR(40003, "订单状态不允许此操作"),
    
    // 系统错误 (500xx)
    SYSTEM_ERROR(50000, "系统繁忙,请稍后重试"),
    DATABASE_ERROR(50001, "数据库操作失败");

    // 错误码 - 用于前后端约定
    private final int code;
    
    // 错误信息 - 用于返回给前端展示
    private final String message;

    /**
     * 枚举构造方法
     * 注意: 枚举的构造方法必须是 private 的(可以省略不写)
     */
    ErrorCode(int code, String message) {
        this.code = code;
        this.message = message;
    }

    /**
     * 获取错误码
     */
    public int getCode() {
        return code;
    }

    /**
     * 获取错误信息
     */
    public String getMessage() {
        return message;
    }
}
```

**为什么要用枚举而不是常量类?**

```java
// ❌ 不推荐: 用常量类
public class ErrorCodeConstants {
    public static final int INVALID_PARAM = 40000;
    public static final String INVALID_PARAM_MSG = "参数不合法";
    // code 和 message 分离,容易出错
}

// ✅ 推荐: 用枚举
public enum ErrorCode {
    INVALID_PARAM(40000, "参数不合法");
    // code 和 message 绑定在一起,不会出错
}
```

#### 第二步:定义业务异常类

```java
/**
 * 业务异常类
 * 
 * 设计要点:
 * 1. 继承 RuntimeException - 不需要在方法签名上声明 throws
 * 2. 包含错误码 - 方便统一异常处理器识别
 * 3. 提供多种构造方法 - 适应不同场景
 * 
 * 继承关系:
 * BizException -> RuntimeException -> Exception -> Throwable
 * 
 * 说明: 当我们在构造方法中使用 super() 时,调用的是直接父类 RuntimeException 的构造方法
 */
public class BizException extends RuntimeException {
    
    // 错误码 - 用于返回给前端
    private final int code;

    /**
     * 构造方法1: 只传错误码枚举
     * 
     * 使用场景: 错误信息不需要额外补充时
     * 例如: throw new BizException(ErrorCode.USER_NOT_FOUND);
     */
    public BizException(ErrorCode errorCode) {
        // 调用父类 RuntimeException 的构造方法,设置异常信息
        // super() 是调用父类构造方法的关键字
        // 这里把 errorCode.getMessage() 传给 RuntimeException,作为异常的描述信息
        super(errorCode.getMessage());
        this.code = errorCode.getCode();
    }

    /**
     * 构造方法2: 传错误码枚举 + 详细信息
     * 
     * 使用场景: 需要补充具体的业务信息时
     * 例如: throw new BizException(ErrorCode.OUT_OF_STOCK, "iPhone 15 Pro");
     */
    public BizException(ErrorCode errorCode, String detail) {
        // 拼接基础信息和详细信息
        super(errorCode.getMessage() + ": " + detail);
        this.code = errorCode.getCode();
    }

    /**
     * 构造方法3: 传错误码枚举 + 详细信息 + 原始异常
     * 
     * 使用场景: 需要保留原始异常堆栈时
     * 例如: catch (SQLException e) {
     *          throw new BizException(ErrorCode.DATABASE_ERROR, "保存用户失败", e);
     *       }
     */
    public BizException(ErrorCode errorCode, String detail, Throwable cause) {
        super(errorCode.getMessage() + ": " + detail, cause);
        this.code = errorCode.getCode();
    }

    /**
     * 获取错误码
     * 全局异常处理器会用这个方法获取错误码
     */
    public int getCode() {
        return code;
    }
}
```

**为什么继承 RuntimeException 而不是 Exception?**

```java
// 如果继承 Exception (受检异常)
public void placeOrder(...) throws BizException {
    // 每个方法都要声明 throws
}

public void processOrder(...) throws BizException {
    placeOrder(...); // 必须处理或继续声明 throws
}

// 如果继承 RuntimeException (非受检异常)
public void placeOrder(...) {
    // 不需要声明 throws
}

public void processOrder(...) {
    placeOrder(...); // 不需要显式处理
}
```

在现代 Java 后端开发中,业务异常通常继承 `RuntimeException`,因为:
1. 代码更简洁,不需要层层声明 `throws`
2. 由全局异常处理器统一处理
3. 更符合"异常是异常情况"的理念

### 7.3 在业务里使用

现在我们来看一个完整的业务场景:用户下单。

```java
import java.math.BigDecimal;

/**
 * 商品信息
 * 使用 record 简化代码(Java 14+)
 */
record Product(String name, int stock, BigDecimal price) {}

/**
 * 订单服务
 */
public class OrderService {

    /**
     * 下单方法
     * 
     * @param userId 用户ID
     * @param product 商品信息
     * @param balance 用户余额
     */
    public void placeOrder(Long userId, Product product, BigDecimal balance) {
        
        // ========== 参数校验 ==========
        // 校验1: 用户ID不能为空
        if (userId == null) {
            // 抛出参数错误异常,并补充具体是哪个参数
            throw new BizException(ErrorCode.INVALID_PARAM, "userId 不能为空");
        }

        // 校验2: 商品信息不能为空
        if (product == null) {
            throw new BizException(ErrorCode.INVALID_PARAM, "商品不能为空");
        }

        // ========== 业务规则校验 ==========
        // 规则1: 检查库存是否充足
        if (product.stock() <= 0) {
            // 抛出库存不足异常,并告知是哪个商品
            throw new BizException(ErrorCode.OUT_OF_STOCK, product.name());
            // 最终异常信息: "库存不足: iPhone 15 Pro"
        }

        // 规则2: 检查余额是否充足
        // BigDecimal.compareTo() 返回值:
        // - 负数: balance < product.price()
        // - 0: balance == product.price()
        // - 正数: balance > product.price()
        if (balance.compareTo(product.price()) < 0) {
            // 抛出余额不足异常,并告知当前余额和商品价格
            String detail = String.format(
                "当前余额 %s 元,商品价格 %s 元",
                balance,
                product.price()
            );
            throw new BizException(ErrorCode.INSUFFICIENT_BALANCE, detail);
            // 最终异常信息: "余额不足: 当前余额 50.00 元,商品价格 99.00 元"
        }

        // ========== 执行下单逻辑 ==========
        // 如果所有校验都通过,执行实际的下单操作
        // 真实项目里这里会:
        // 1. 扣减库存
        // 2. 扣减余额
        // 3. 创建订单记录
        // 4. 发送消息通知
        System.out.println("下单成功: " + product.name());
    }
}
```

#### 使用示例


### 7.4 为什么这比字符串异常更好？

因为你拿到异常时，可以同时得到：

- 业务分类
- 错误码
- 对前端友好的提示
- 更容易统一响应格式

这在全栈协作里很重要。前端最怕的不是报错，而是：

- 后端有时候返回 `"失败了"`
- 有时候返回 `"error"`
- 有时候直接 500 没文案

自定义异常就是把错误表达标准化。

## 八、后端实战里，异常应该怎么落地？

这一段非常重要，因为它解决的是“会写 `try/catch`，但不知道该写在哪”的问题。

### 8.1 推荐的分工方式

| 层 | 推荐做法 |
|------|------|
| 参数校验层 | 参数错了就尽早报错，不要拖到深层逻辑 |
| Service 层 | 负责业务规则，适合抛 `BizException` |
| Repository 层 | 负责访问数据库，通常不拼前端提示文案 |
| 全局异常处理器 | 统一转成 `Result<T>` 或标准错误 JSON |

### 8.2 一个非常典型的思路

```text
Controller 收到请求
  -> Service 发现库存不足
  -> throw BizException
  -> 全局异常处理器捕获
  -> 返回 { code, message, data: null }
```

你可以把它类比成前端里的：

```text
页面发请求
  -> axios 收到错误响应
  -> response interceptor 统一提示
```

### 8.3 什么时候 catch，什么时候继续抛？

可以用这个标准判断：

#### 情况 1：你有能力恢复流程

那就 catch。

例如：

- 读取配置文件失败，改用默认配置
- 某个三方接口失败，走降级方案

#### 情况 2：你没有能力恢复，只知道出错了

那就不要硬接，继续抛。

例如：

- 用户不存在
- 订单状态非法
- 数据库写入失败，当前层也无能为力

### 8.4 日志怎么打才合理？

一个常见错误是：每一层都 `log.error()` 一遍，最后一条异常打了 5 次。

更合理的原则：

1. 关键上下文要打日志
2. 不要重复打同一条异常
3. 对用户返回友好文案，对日志保留技术细节

例如：

```java
try {
    orderService.placeOrder(1L, product, balance);
} catch (BizException e) {
    log.warn("下单失败, userId={}, code={}, message={}", 1L, e.getCode(), e.getMessage());
    throw e;
} catch (Exception e) {
    log.error("系统异常, userId={}", 1L, e);
    throw new BizException(ErrorCode.SYSTEM_ERROR);
}
```

上面只是演示思路。真实项目里，很多时候你会把统一处理放在全局异常处理器里，而不是每个 Controller 手写。

### 8.5 常见反模式

#### 反模式 1：吞异常

```java
try {
    // ...
} catch (Exception e) {
}
```

问题：

- 调用方以为成功了
- 日志里没有证据
- Bug 会变得很难查

#### 反模式 2：拿异常当流程控制

```java
Integer age;

try {
    age = Integer.parseInt(input);
} catch (Exception e) {
    age = 18;
}
```

更合理的是先做校验，而不是故意让它异常。

#### 反模式 3：抛新异常但丢掉原始原因

```java
catch (Exception e) {
    throw new RuntimeException("保存失败");
}
```

这样会丢失根因。

正确一点：

```java
catch (Exception e) {
    throw new RuntimeException("保存失败", e);
}
```

#### 反模式 4：把技术异常直接返回给前端

比如把数据库异常原样返回：

```text
SQLSyntaxErrorException: You have an error in your SQL syntax ...
```

这既不友好，也可能泄露系统细节。

## 九、泛型是什么？先用前端思维理解它

### 9.1 一句话解释

**泛型就是“把类型也参数化”。**

你以前给方法传参数，现在 Java 允许你给类、接口、方法传“类型参数”。

### 9.2 和 TypeScript 泛型的对应关系

| TypeScript | Java |
|------|------|
| `type ApiResult<T> = { data: T }` | `class Result<T> { private T data; }` |
| `function identity<T>(x: T): T` | `public static <T> T identity(T x)` |
| `Array<User>` | `List<User>` |
| `Record<string, User>` | `Map<String, User>` |

如果你学过 TS，你会对 Java 泛型有天然优势。

### 9.3 但 Java 泛型和 TS 泛型并不完全一样

这里要特别注意：

1. Java 泛型主要在编译期检查，运行时会有“类型擦除”
2. Java 泛型只能用引用类型，不能直接用基本类型
3. Java 在集合泛型上通常比 TS 更严格
4. Java 有 `? extends T`、`? super T` 这种通配符写法，初学时容易懵

例如：

```java
List<Integer> numbers = new ArrayList<>();
```

这里必须写 `Integer`，不能写 `int`。

## 十、泛型基础：为什么它能让后端代码更安全？

### 10.1 没有泛型时的问题

```java
import java.util.ArrayList;
import java.util.List;

public class RawTypeDemo {
    public static void main(String[] args) {
        List list = new ArrayList();
        list.add("hello");
        list.add(123);

        String first = (String) list.get(0);
        String second = (String) list.get(1); // 运行时报错
        System.out.println(first + second);
    }
}
```

问题：

- 什么都能放
- 取出来要强转
- 错误延迟到运行时才暴露

### 10.2 有泛型后的写法

```java
import java.util.ArrayList;
import java.util.List;

public class GenericListDemo {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        list.add("hello");
        // list.add(123); // 编译错误

        String first = list.get(0);
        System.out.println(first);
    }
}
```

好处：

1. 编译期就拦住错误
2. 不用强制类型转换
3. 方法签名更清晰
4. IDE 自动提示更准确

### 10.3 泛型类

```java
public class Box<T> {
    private T content;

    public void put(T content) {
        this.content = content;
    }

    public T get() {
        return content;
    }

    public boolean isEmpty() {
        return content == null;
    }
}
```

使用：

```java
record User(Long id, String name) {}

public class BoxDemo {
    public static void main(String[] args) {
        Box<String> stringBox = new Box<>();
        stringBox.put("hello");
        String text = stringBox.get();

        Box<User> userBox = new Box<>();
        userBox.put(new User(1L, "张三"));
        User user = userBox.get();
    }
}
```

你可以把 `Box<T>` 理解成：

- `Box<String>`：只能装字符串
- `Box<User>`：只能装用户对象

### 10.4 泛型方法

泛型不只可以放在类上，也可以放在方法上。

```java
public class GenericMethodDemo {

    public static <T> T pickFirst(T[] array) {
        if (array == null || array.length == 0) {
            throw new IllegalArgumentException("数组不能为空");
        }
        return array[0];
    }

    public static void main(String[] args) {
        String[] words = {"java", "spring"};
        Integer[] nums = {1, 2, 3};

        String firstWord = pickFirst(words);
        Integer firstNum = pickFirst(nums);

        System.out.println(firstWord);
        System.out.println(firstNum);
    }
}
```

这里的核心是：

- 方法只写一份
- 编译器会根据调用时的参数推导类型

### 10.5 带边界的泛型

有时候你不是“任意类型都行”，而是“必须具备某种能力”。

比如找最大值，要求元素必须能比较大小：

```java
public class MaxDemo {

    public static <T extends Comparable<T>> T max(T a, T b) {
        return a.compareTo(b) >= 0 ? a : b;
    }

    public static void main(String[] args) {
        System.out.println(max(3, 5));
        System.out.println(max("apple", "banana"));
    }
}
```

`T extends Comparable<T>` 的意思是：

这个 `T` 必须实现 `Comparable<T>`，否则不能调用 `compareTo`。

## 十一、泛型进阶：通配符、边界、PECS

这一段是很多人的难点，但其实抓住一个核心就够了：

**你是要“读”为主，还是要“写”为主？**

### 11.1 为什么需要通配符？

如果你写了这样一个方法：

```java
public static void printList(List<String> list) {
    for (String item : list) {
        System.out.println(item);
    }
}
```

它只能接收 `List<String>`。

那如果你还想打印：

- `List<Integer>`
- `List<User>`
- `List<Product>`

总不能每种都重载一遍，所以就要用通配符。

### 11.2 `<?>`：无界通配符

```java
import java.util.List;

public class WildcardDemo {

    public static void printList(List<?> list) {
        for (Object item : list) {
            System.out.println(item);
        }
    }
}
```

这表示：**任意类型的 `List` 都可以传进来**。

但注意：

- 你可以安全地读出来，当作 `Object`
- 你不能随便往里放元素

因为编译器并不知道它到底是 `List<String>` 还是 `List<Integer>`。

### 11.3 `<? extends T>`：上界通配符

```java
import java.util.List;

public class ExtendsDemo {

    public static double sum(List<? extends Number> list) {
        double total = 0;
        for (Number number : list) {
            total += number.doubleValue();
        }
        return total;
    }
}
```

这表示：参数可以是：

- `List<Integer>`
- `List<Double>`
- `List<Long>`

因为它们都是 `Number` 的子类。

特点：

- 适合“读”
- 不适合“写”

你能从里面安全地读成 `Number`，但不能安全地 `add` 一个元素。

### 11.4 `<? super T>`：下界通配符

```java
import java.util.List;

public class SuperDemo {

    public static void addDefaultValues(List<? super Integer> list) {
        list.add(1);
        list.add(2);
        list.add(3);
    }
}
```

这表示参数可以是：

- `List<Integer>`
- `List<Number>`
- `List<Object>`

特点：

- 适合“写”
- 读出来时只能保守地当 `Object`

### 11.5 PECS 原则

你只需要记这句：

```text
Producer Extends, Consumer Super
```

翻译成人话：

- 如果一个集合主要是“生产数据给你读” -> `extends`
- 如果一个集合主要是“接收你写入的数据” -> `super`

### 11.6 为什么 `List<Object>` 不能接收 `List<String>`？

这是 Java 泛型里一个特别经典的问题。

很多人会想当然地以为：

```java
Object 是 String 的父类
所以 List<Object> 应该也是 List<String> 的父类
```

这是错的。

看这个例子：

```java
List<String> strings = new ArrayList<>();
// List<Object> objects = strings; // 编译错误
```

为什么不能这样？

因为如果允许：

```java
List<Object> objects = new ArrayList<>();
objects.add(123);
```

那原来的 `strings` 里就会混进 `Integer`，类型安全直接崩了。

所以 Java 泛型默认是 **不协变** 的。你可以简单理解为：

**`List<String>` 和 `List<Object>` 是两个完全不同的类型。**

这也是为什么通配符会存在。

## 十二、泛型擦除与常见坑

### 12.1 什么叫类型擦除？

Java 泛型大部分只存在于编译期。到了运行期，很多具体泛型信息会被擦掉。

```java
import java.util.ArrayList;
import java.util.List;

public class ErasureDemo {
    public static void main(String[] args) {
        List<String> strings = new ArrayList<>();
        List<Integer> integers = new ArrayList<>();

        System.out.println(strings.getClass() == integers.getClass()); // true
    }
}
```

运行结果是 `true`，说明运行时它们本质上都是 `ArrayList`。

### 12.2 这会带来什么影响？

#### 不能直接创建泛型数组

```java
// T[] array = new T[10]; // 编译错误
```

#### 不能直接 `new T()`

```java
// T obj = new T(); // 编译错误
```

#### 不能用基本类型做泛型参数

```java
// List<int> nums = new ArrayList<>(); // 编译错误
List<Integer> nums = new ArrayList<>();
```

### 12.3 新手阶段你只要记住这些就够了

1. 泛型主要帮你做编译期类型检查
2. 运行时不要指望它保留所有类型信息
3. `List<int>` 不行，写 `List<Integer>`
4. 遇到 `? extends`、`? super` 先判断“我要读还是写”

## 十三、综合实战：统一响应 + 分页 + 全局异常思路

这一节把异常和泛型串起来，让你感受到它们在后端里为什么经常一起出现。

### 13.1 前端最熟悉的返回结构

很多前端项目里都会有这种类型：

```ts
type Result<T> = {
  code: number
  message: string
  data: T | null
}

type PageResult<T> = {
  list: T[]
  page: number
  pageSize: number
  total: number
}
```

在 Java 后端里，通常就会落成泛型类。

### 13.2 `PageResult<T>`

```java
import java.util.List;

public class PageResult<T> {
    private final List<T> list;
    private final int page;
    private final int pageSize;
    private final long total;

    public PageResult(List<T> list, int page, int pageSize, long total) {
        this.list = list;
        this.page = page;
        this.pageSize = pageSize;
        this.total = total;
    }

    public List<T> getList() {
        return list;
    }

    public int getPage() {
        return page;
    }

    public int getPageSize() {
        return pageSize;
    }

    public long getTotal() {
        return total;
    }
}
```

### 13.3 `Result<T>`

```java
public class Result<T> {
    private final int code;
    private final String message;
    private final T data;

    private Result(int code, String message, T data) {
        this.code = code;
        this.message = message;
        this.data = data;
    }

    public static <T> Result<T> success(T data) {
        return new Result<>(200, "success", data);
    }

    public static <T> Result<T> error(int code, String message) {
        return new Result<>(code, message, null);
    }

    public int getCode() {
        return code;
    }

    public String getMessage() {
        return message;
    }

    public T getData() {
        return data;
    }
}
```

### 13.4 业务代码里如何配合异常使用

```java
import java.util.List;

record UserVO(Long id, String name) {}

public class UserService {

    public Result<PageResult<UserVO>> listUsers(int page, int pageSize) {
        if (page < 1 || pageSize < 1) {
            throw new BizException(ErrorCode.INVALID_PARAM, "page 和 pageSize 必须大于 0");
        }

        List<UserVO> users = List.of(
            new UserVO(1L, "张三"),
            new UserVO(2L, "李四")
        );

        PageResult<UserVO> pageResult = new PageResult<>(users, page, pageSize, 2);
        return Result.success(pageResult);
    }
}
```

这里的关键点有两个：

1. **泛型** 让 `Result<T>` 和 `PageResult<T>` 可以复用在任何数据类型上
2. **异常** 让非法参数、业务失败从正常流程里分离出去

### 13.5 如果到了 Spring Boot，最终通常会变成这样

下面是预览，你现在先看懂思路就行，后面实战篇还会正式讲。

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BizException.class)
    public Result<Void> handleBizException(BizException e) {
        return Result.error(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public Result<Void> handleException(Exception e) {
        return Result.error(50000, "系统繁忙，请稍后重试");
    }
}
```

这就相当于后端的“统一错误出口”。

前端收到的响应会更稳定，比如：

```json
{
  "code": 40000,
  "message": "参数不合法: page 和 pageSize 必须大于 0",
  "data": null
}
```

这也是为什么后面那篇 [05-参数校验与统一异常响应](../04-企业级工程化篇/05-参数校验与统一异常响应.md) 会非常重要。

## 十四、面向前端转全栈的常见困惑

### 14.1 “我是不是应该到处写 `try/catch`？”

不是。

很多初学者会误以为后端就是“哪里可能出错，哪里包一个 `try/catch`”。

实际上更好的思路是：

- 有恢复方案才 catch
- 没恢复方案就上抛
- 在统一边界做最终响应转换

### 14.2 “业务异常和系统异常怎么区分？”

你可以先粗暴但实用地这样分：

| 类型 | 例子 |
|------|------|
| 业务异常 | 用户不存在、库存不足、余额不足、订单状态非法 |
| 系统异常 | 空指针、数据库连接失败、网络超时、文件损坏 |

业务异常往往可以明确告诉前端；系统异常通常只返回统一提示，不暴露内部细节。

### 14.3 “Java 泛型和 TS 泛型谁更难？”

如果你有 TS 基础，前 70% 不难。

真正容易卡住的是后 30%：

- `? extends T`
- `? super T`
- 类型擦除
- `List<Object>` 和 `List<String>` 不是父子关系

但好消息是，日常业务开发里最常写的其实还是：

```java
List<User>
Map<String, Object>
Result<UserVO>
PageResult<OrderVO>
```

先把这些写熟，再去吃透通配符。

## 十六、本章总结

如果你是前端转全栈，这一章最应该带走的是下面 6 句话：

1. 异常不是“报错提示”，而是“程序流程出现异常分支”。
2. `throw` 是抛出异常，`throws` 是声明可能抛异常。
3. 不是所有地方都要 `catch`，只有你能处理时才处理。
4. 业务异常通常会自定义，并在统一边界转成标准响应。
5. 泛型的核心价值是复用代码，同时保证类型安全。
6. 日常业务开发里最常见的泛型写法就是 `List<T>`、`Map<K, V>`、`Result<T>`、`PageResult<T>`。

再压缩成两句：

- **异常处理**：想清楚“哪里抛、哪里接、最终怎么返回给前端”。  
- **泛型**：想清楚“这份代码是不是可以服务多种类型，而且还能保证不写错类型”。  

如果你把这两件事理解透，后面学 Spring Boot、统一异常响应、分页接口、通用返回体时会轻松很多。
