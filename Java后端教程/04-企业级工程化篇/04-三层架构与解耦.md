> **本章目标**：深入理解 Controller → Service → Repository 三层架构，掌握依赖倒置和可测试性设计  
> **预计耗时**：2~3 小时

## 一、为什么要分层？

### 1.1 不分层的代码（反面教材）

```java
// ❌ 所有逻辑堆在 Controller 里，像一锅乱炖
@PostMapping("/orders")
public Result<Order> createOrder(@RequestBody OrderDTO dto) {
    // 1. 参数校验
    if (dto.getProductId() == null) throw new RuntimeException("商品ID不能为空");

    // 2. 查数据库
    Product product = entityManager.find(Product.class, dto.getProductId());
    if (product == null) throw new RuntimeException("商品不存在");

    // 3. 业务逻辑
    if (product.getStock() < dto.getQuantity()) throw new RuntimeException("库存不足");
    product.setStock(product.getStock() - dto.getQuantity());
    entityManager.merge(product);

    // 4. 创建订单
    Order order = new Order();
    order.setProductId(product.getId());
    order.setQuantity(dto.getQuantity());
    order.setTotalPrice(product.getPrice() * dto.getQuantity());
    entityManager.persist(order);

    // 5. 发通知
    // ... 发邮件、发短信的代码也塞在这里

    return Result.success(order);
}
```

**问题**：
- Controller 一个方法几百行，难以阅读
- 修改业务逻辑可能影响接口层
- 代码无法复用（其他地方也要下单怎么办？再写一遍？）
- 无法单独测试业务逻辑

### 1.2 分层后的代码

```java
// ✅ Controller: 只做参数接收和返回
@PostMapping("/orders")
public Result<OrderVO> createOrder(@Valid @RequestBody CreateOrderRequest req) {
    Order order = orderService.createOrder(req);
    return Result.success(OrderVO.fromEntity(order));
}
```

```java
// ✅ Service: 只做业务逻辑
@Service
@RequiredArgsConstructor
public class OrderService {

    private final ProductRepository productRepository;
    private final OrderRepository orderRepository;
    private final NotificationService notificationService;

    @Transactional
    public Order createOrder(CreateOrderRequest req) {
        Product product = productRepository.findById(req.getProductId())
            .orElseThrow(() -> BizException.notFound("商品不存在"));

        if (product.getStock() < req.getQuantity()) {
            throw BizException.badRequest("库存不足");
        }

        product.setStock(product.getStock() - req.getQuantity());
        productRepository.save(product);

        Order order = new Order();
        order.setProduct(product);
        order.setQuantity(req.getQuantity());
        order.setTotalPrice(product.getPrice() * req.getQuantity());
        order.setStatus(OrderStatus.PENDING);
        Order saved = orderRepository.save(order);

        notificationService.sendOrderNotification(saved);

        return saved;
    }
}
```

```java
// ✅ Repository: 只做数据访问
@Repository
public interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByUserIdOrderByCreatedAtDesc(Long userId);
}
```

## 二、三层的职责边界

```
┌─────────────────────────────────────────────┐
│  Controller 层（薄！越薄越好）                │
│  ✅ 接收参数、参数校验(@Valid)                 │
│  ✅ 调用 Service 方法                         │
│  ✅ 组装返回值 (Entity → VO)                  │
│  ❌ 不写 if/else 业务判断                      │
│  ❌ 不直接调用 Repository                      │
│  ❌ 不操作 HttpSession                         │
├─────────────────────────────────────────────┤
│  Service 层（厚！核心逻辑都在这里）            │
│  ✅ 业务逻辑（校验、计算、流程编排）           │
│  ✅ 事务管理(@Transactional)                   │
│  ✅ 调用多个 Repository 协同操作               │
│  ✅ 调用其他 Service                           │
│  ❌ 不接触 HTTP 请求/响应对象                   │
│  ❌ 不返回 Result 包装（由 Controller 包装）    │
├─────────────────────────────────────────────┤
│  Repository 层（数据操作）                     │
│  ✅ 数据库 CRUD                                │
│  ✅ 自定义查询 SQL                             │
│  ❌ 不写业务逻辑                               │
│  ❌ 不做数据转换                               │
└─────────────────────────────────────────────┘
```

## 三、依赖方向：只能从上往下

```
Controller → Service → Repository → Database
     ↓           ↓          ↓
   依赖        依赖       依赖

❌ Repository 不能调用 Service
❌ Service 不能调用 Controller
❌ 不能有循环依赖（A 依赖 B，B 又依赖 A）
```

如果发现循环依赖，说明设计有问题，应该：
- 提取公共逻辑到新的 Service
- 或者用事件机制解耦

## 四、依赖倒置：面向接口编程

### 4.1 为什么要用接口？

```java
// ❌ 直接依赖具体实现：如果要换通知方式（邮件→短信），Service 要改代码
@Service
public class OrderService {
    private final EmailSender emailSender; // 直接依赖邮件发送器

    public void createOrder(...) {
        emailSender.send(...); // 写死了用邮件通知
    }
}

// ✅ 依赖接口：随时切换实现，Service 不用改
public interface NotificationSender {
    void send(String to, String message);
}

@Service
public class OrderService {
    private final NotificationSender sender; // 依赖接口

    public void createOrder(...) {
        sender.send(...); // 不关心是邮件还是短信
    }
}

// 实现类：邮件通知
@Component
public class EmailNotificationSender implements NotificationSender {
    public void send(String to, String message) {
        // 发邮件
    }
}

// 实现类：短信通知（想换就换，创建新类即可）
@Component
public class SmsNotificationSender implements NotificationSender {
    public void send(String to, String message) {
        // 发短信
    }
}
```

### 4.2 多个实现如何选择？

```java
// 当有多个实现时，用 @Qualifier 指定
@Service
public class OrderService {

    private final NotificationSender sender;

    public OrderService(@Qualifier("emailNotificationSender") NotificationSender sender) {
        this.sender = sender;
    }
}

// 或者在实现类上用 @Primary 标记默认实现
@Primary
@Component
public class EmailNotificationSender implements NotificationSender { ... }
```

## 五、可测试性

分层架构最大的好处之一：**每一层都可以独立测试**。

```java
// 测试 Service 时，不需要启动整个 Spring 应用
// 也不需要真正的数据库
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private ProductRepository productRepository;

    @Mock
    private OrderRepository orderRepository;

    @InjectMocks
    private OrderService orderService;

    @Test
    void shouldCreateOrderSuccessfully() {
        // 准备模拟数据
        Product product = new Product();
        product.setId(1L);
        product.setPrice(99.9);
        product.setStock(10);

        when(productRepository.findById(1L)).thenReturn(Optional.of(product));
        when(orderRepository.save(any())).thenAnswer(inv -> inv.getArgument(0));

        // 执行测试
        CreateOrderRequest req = new CreateOrderRequest();
        req.setProductId(1L);
        req.setQuantity(2);

        Order order = orderService.createOrder(req);

        // 验证结果
        assertEquals(199.8, order.getTotalPrice(), 0.01);
        assertEquals(8, product.getStock()); // 库存应该减了2
        verify(orderRepository).save(any());
    }

    @Test
    void shouldThrowWhenOutOfStock() {
        Product product = new Product();
        product.setStock(1);

        when(productRepository.findById(1L)).thenReturn(Optional.of(product));

        CreateOrderRequest req = new CreateOrderRequest();
        req.setProductId(1L);
        req.setQuantity(5); // 库存只有1，要买5

        assertThrows(BizException.class, () -> orderService.createOrder(req));
    }
}
```

## 六、常见反模式（要避免的）

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| Controller 里写业务逻辑 | 难维护、难复用 | 移到 Service |
| Service 返回 Result | 耦合了 Web 响应格式 | Service 返回原始数据，Controller 包 Result |
| Controller 直接调 Repository | 跳过了业务层 | 通过 Service 调用 |
| Service A 和 Service B 循环依赖 | Spring 启动报错 | 提取公共 Service C |
| Entity 直接返回给前端 | 暴露敏感字段、耦合 | 转换为 VO 再返回 |

## 七、本章总结

| 要点 | 说明 |
|------|------|
| 三层职责 | Controller 薄（接收参数）、Service 厚（业务逻辑）、Repository（数据操作） |
| 依赖方向 | 上层依赖下层，不能反向依赖 |
| 面向接口 | Service 依赖接口不依赖具体实现，方便扩展 |
| 可测试性 | 分层后每层可独立测试（Mock 依赖） |
| DTO 分离 | Entity/DTO/VO 各司其职，不混用 |
