# 01 - 标准项目目录结构（零基础详解版）

> **本章目标**：理解企业级 Spring Boot 项目的标准目录结构，知道每个目录和文件的作用  
> **预计耗时**：1~2 小时  
> **前置要求**：完成实战篇

---

## 一、为什么需要标准目录结构？

### 1.1 生活类比

```
没有规范的项目 = 你的房间：
├── 袜子丢在厨房
├── 书本放在卫生间
├── 外卖盒堆在床上
└── 找什么都找不到

有规范的项目 = 宜家样板间：
├── 客厅放沙发电视（展示层 Controller）
├── 厨房做饭（业务逻辑 Service）
├── 冰箱放食材（数据层 Repository）
├── 储物间放杂物（工具类 Utils）
└── 每样东西都有固定的位置
```

**好处**：
- 新人加入团队能快速找到代码
- 不同模块职责清晰，不会互相干扰
- 代码可维护性大大提高

## 三、每一层的职责边界

### 3.1 请求处理流程

```
客户端请求
    │
    ▼
Filter（过滤器）     ← 最外层：CORS、编码、TraceID
    │
    ▼
Interceptor（拦截器） ← 登录检查、权限验证
    │
    ▼
Controller（控制器）  ← 接收参数、参数校验、调用 Service、返回结果
    │                    ❌ 不要写业务逻辑！
    ▼
Service（服务层）     ← 核心业务逻辑、事务管理
    │                    ❌ 不要直接操作 HTTP 请求/响应！
    ▼
Repository（数据层）  ← 操作数据库（CRUD）
    │                    ❌ 不要写业务逻辑！
    ▼
Database（数据库）
```

### 3.2 各层规则

| 层 | 能做什么 | 不能做什么 |
|---|---------|-----------|
| **Controller** | 接收参数、校验、调用 Service、返回 Result | 写业务逻辑、直接调 Repository |
| **Service** | 业务逻辑、事务、调用 Repository | 操作 HttpServletRequest、返回 HTTP 状态码 |
| **Repository** | 数据库操作 | 写业务逻辑 |
| **DTO** | 定义接口的数据格式 | 包含业务逻辑方法 |
| **Entity** | 映射数据库表 | 直接暴露给前端 |

### 3.3 DTO 和 Entity 分离的重要性

```
❌ 错误做法：直接把 Entity 返回给前端
  → 密码字段也返回了！
  → 数据库字段变动直接影响前端！
  → 前端看到了不需要的字段

✅ 正确做法：用 DTO 做数据转换
  Entity（数据库）→ 转换 → VO/DTO（返回给前端）
  前端请求 DTO → 转换 → Entity（存入数据库）
```

```java
// Entity: 数据库表映射（包含敏感字段）
@Entity
public class User {
    private Long id;
    private String username;
    private String password;   // 密码！不能返回给前端
    private String email;
    private Boolean enabled;
    private LocalDateTime createdAt;
}

// VO: 返回给前端的数据（排除敏感信息）
@Data
public class UserVO {
    private Long id;
    private String username;
    // 没有 password！
    private String email;
    private LocalDateTime createdAt;

    // Entity → VO 的转换方法
    public static UserVO fromEntity(User user) {
        UserVO vo = new UserVO();
        vo.setId(user.getId());
        vo.setUsername(user.getUsername());
        vo.setEmail(user.getEmail());
        vo.setCreatedAt(user.getCreatedAt());
        return vo;
    }
}
```

## 四、按层分包 vs 按领域分包

### 4.1 按层分包（推荐新手使用）

上面介绍的就是按层分包，**按代码的技术职责分目录**：

```
com.example.demo/
├── controller/    ← 所有 Controller 放在一起
├── service/       ← 所有 Service 放在一起
├── repository/    ← 所有 Repository 放在一起
└── entity/        ← 所有 Entity 放在一起
```

优点：结构清晰，新手友好
缺点：项目大了以后，一个目录下文件太多

### 4.2 按领域分包（大项目推荐）

```
com.example.demo/
├── user/              ← 用户模块（所有用户相关放一起）
│   ├── UserController.java
│   ├── UserService.java
│   ├── UserRepository.java
│   ├── User.java
│   └── dto/
│       ├── CreateUserRequest.java
│       └── UserVO.java
│
├── order/             ← 订单模块
│   ├── OrderController.java
│   ├── OrderService.java
│   ├── OrderRepository.java
│   ├── Order.java
│   └── dto/
│
├── product/           ← 商品模块
│   └── ...
│
└── common/            ← 公共模块
    ├── Result.java
    ├── BizException.java
    └── ...
```

优点：高内聚（相关代码放一起），适合大项目和微服务拆分
缺点：初期可能觉得目录层级多

## 五、命名规范

| 类型 | 命名规则 | 示例 |
|------|---------|------|
| 包名 | 全小写 | `com.example.demo.controller` |
| 类名 | 大驼峰 | `UserController`, `OrderService` |
| 方法名 | 小驼峰 | `findById`, `createOrder` |
| 变量名 | 小驼峰 | `userName`, `orderList` |
| 常量 | 全大写下划线 | `MAX_PAGE_SIZE`, `DEFAULT_TIMEOUT` |
| 配置文件 key | 小写中划线 | `spring.data-source.url` |
| 数据库表名 | 小写下划线 | `user_orders`, `product_categories` |
| REST 路径 | 小写中划线 | `/api/user-orders` |

### 常见类命名后缀

| 后缀 | 说明 | 示例 |
|------|------|------|
| Controller | 控制器 | `UserController` |
| Service | 服务层 | `UserService` |
| Repository | 数据层 | `UserRepository` |
| DTO | 数据传输对象 | `CreateUserDTO` |
| VO | 视图对象（返回给前端） | `UserVO` |
| Config | 配置类 | `RedisConfig` |
| Filter | 过滤器 | `CorsFilter` |
| Handler | 处理器 | `GlobalExceptionHandler` |
| Utils / Helper | 工具类 | `DateUtils` |
| Enum | 枚举 | `OrderStatus` |

## 六、本章总结

| 要点 | 说明 |
|------|------|
| 分层架构 | Controller → Service → Repository，职责清晰 |
| DTO 分离 | Entity 不直接暴露给前端，用 DTO/VO 做转换 |
| 新手建议 | 先用"按层分包"，项目变大后再考虑"按领域分包" |
| 命名规范 | 统一命名风格，提高可读性 |
